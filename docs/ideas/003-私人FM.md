# 003 - 私人FM

> 开始：2026-07-15（002 歌词对齐归档后的新阶段）

## 🎯 这个阶段的产物

Biu 有「私人FM」——网易云「心动模式」的复刻。从默认不可删的红心歌单
「我喜欢的音乐」取种子，用 B站原生信号 + 网易相似表找相似**单曲**，与红心
歌交织、无限续供；进入即自动播放、跨会话不重复推。从任意曲库歌单
（收藏夹 / 本地 / 合集 / 系列）点歌自动切走 FM 并整队替换成该歌单，全网
搜索点歌则插播保留 FM。用户往红心多加歌 = 推荐越准。

数据流图见 [`DEVLOG.md`](../../DEVLOG.md)（`docs/devlog-assets/private-fm-pipeline.svg`、`netease-leg.svg`）。

## 📖 详细说明

### 关键决策

**「相似」怎么造**：
- 网易的「相似」本质是靠海量听歌行为算好的一张**协同过滤预计算查表**——那张
  表我们造不出，但能借。
- Phase 1 用 B站原生信号自己凑：**看了又看**（`archive/related`，跨 UP、多样性
  高，优先）+ **同 UP 投稿**（`space/arc/search`，搬运号多是单一曲风频道，垫后）。
- Phase 2 借网易 `simiSong` 表当第三条腿（跨歌手、同曲风），排在候选池最前。

**单曲净化两段**（整个功能的地基）：
- Stage 1 纯函数 `isPureSongCandidate`：时长窗 `75~420s` + 标题黑名单 + 子分区
  黑名单，有 `tests/pure-song.test.ts` 兜底。
- Stage 2 `view` 核验：`videos === 1` 剔多 P（合集 / 整张专辑），并用精确时长 /
  子分区复筛一遍 Stage 1。

**同名去重用 `isSameSong` 而非 exact-key 相等**：短串 ≥4 字且被长串包含即视为
同一首，才能把「`[中字] カレンダーガール`」和「`カレンダーガール (Calendar Girl)
— 星宮莓…`」折叠成一首。

**交织必须用 Loop 模式**：相似打底、每 `SIMILAR_PER_LIKED=8` 首插 1 红心，保证
队列 ≥1 红心。用 Loop 播放保住穿插顺序——Random 会打乱穿插比例。

**续供**：订阅 play-list，剩余 < `TOPUP_THRESHOLD=8` 追加一批；**二度扩展种子 = 你在
FM 里收藏的 FM 推荐歌**（强正反馈闭环：FM 推了、你还收藏 → 顺着它找更多同类）。判定门槛
在 store 的 `noteFavoriteFromFm`：**① FM 正在播放 + ② 收藏的歌在 `servedBvids` 里**（红心
被排除在候选外、不进 servedBvids，所以天然只认「FM 推的、非红心的」歌）；命中才 `recordFavoritedSong`
存进 `HeartbeatFavSeeds`（持久、滚动 `MAX_FAV_SEEDS=60`）。收藏动作挂在收藏弹窗 confirm，
不看收进本地还是 B站在线收藏夹（都能覆盖）。续供优先用这批当种子（全元数据 → 喂三腿），
**不够 `SECOND_DEGREE_SEEDS=3` 个时，从最近 40 已推候选随机抽补齐剩余名额**（弱路径兜底，
仅带 bvid → 只喂看了又看）；按红心里 UP 出现频次 `ownerScore` 反馈重排。

**种子轮转覆盖全部红心**：红心超过 `MAX_SEEDS` 时游标轮转（`nextSeeds`），不是
只取前 6 首。

**停 FM 双机制**（这是「歌单点歌切走 FM」正确的关键）：
- **整队替换**（`playList`，即「播放全部」/ 切歌单）→ play-list 内部 id 全变 →
  续供订阅发现本会话 id 一个都不在 → 自动 `stop()`。**架构级兜底，新歌单页
  自动生效，不用逐页接**。
- **单曲点击**（`play` 插播、不删队列）→ 本会话 id 还在 → 兜底不触发 → 必须
  **显式** `stopAndReplace`。这恰是「全网搜索点歌 = 插播保留 FM」需要的行为，
  所以歌单页单曲点击得显式区分，收口进 `service/heartbeat/play-from-folder.ts`
  的 `playFromFolder`，四个子组件（collections / series 各 list+grid）统一走它。

**会话跨重启：接着放，而非重开一轮**（`restoreSession()`；数据流图见 DEVLOG）：

**真根因（藏得深，实测才逮到）——Electron 端持久化根本没通**：`platform.setStore/getStore`
在 Electron 走 `electron/ipc/store.ts` 的**显式白名单 if-链**，而心动三个 key
（`HeartbeatServed / FavSeeds / Session`）从来没登记——`electron/store.ts` 没建 `Store`、IPC 没分支。
于是 `setStore` 静默丢弃、`getStore` 返回 `undefined`，`persistSession` 写了个寂寞、`loadSession`
恒 `active:false` → 恢复永远被拒。**当初加心动时漏了 CLAUDE.md 说的「IPC 契约 5 处」里的存储这处**。
修：补三个 `Store` 实例 + get/set/clear 分支。⚠️ 顺带修好一个潜伏已久的问题——`HeartbeatServed`
（跨会话去重）、`HeartbeatFavSeeds`（收藏种子）自引入起在 Electron 上就一直没真正落地，
只是「偶尔重复推荐」不易察觉。Android 端 `platform` 是通用 `Preferences`（无白名单），一直是好的，
**此 bug 仅 Electron**。

**表层设计**：`useHeartbeat` 纯内存、play-list 持久化——两边寿命不一样，不存会话重启就错位：
队列还在放但 `active/sessionIds` 归零、续供订阅没挂（`attachTopup` 原本只在 `start()` 调）→
**看着是 FM、实则续供已死**的死循环。恢复三件套缺一不可：① `active`+`sessionIds`
② 重挂 `attachTopup()` ③ 载回 `servedBvids/servedKeys`（不载回续供会重复推听过的）。

**隐患清单（都已处理）**：
- **竞态：FM 页判定抢在恢复前**。`restoreSession` 是 async（两次 electron-store IPC），FM 页挂载的
  `isSessionLive()` 是同步。冷启虽默认落 `/`（主窗 `loadFile` 不带 hash，不会直接进 `/heartbeat`），
  但「冷启后极快点 FM」理论上能抢跑 → 误判不在场而重开。修：FM 页 `await restoreSession()` 后再判；
  `restoreSession` 幂等（`if(active)return` + 只读/幂等写），PlayBar 与页面两处并发也安全
  （`attachTopup` 先 `detach`，不会重复订阅）。
- **只能在主窗触发**（PlayBar 只主窗挂载；FM 路由也在主窗布局内）。三窗共用一个 bundle，
  挂模块级会让 mini-player / 桌面歌词各挂一个续供订阅 → 队列见底时重复抓取、重复入队。
- **落盘只存队列里还在的 id**（`sessionIds` 随会话累积），`start/stop/maybeTopup` 三处状态变更都落盘；
  `stop()` 写 `active:false` 自愈——下次重启不误恢复。
- **`ownerScore` 不进持久化**：只有 `maybeTopup` 一个读取点，那儿本就调 `likedContext()`，就地取新的即可
  ——省一个异步依赖，顺带让会话期间新加的红心即时生效。
- **水合时机**：play-list 走默认 localStorage（**同步水合**），读 `list` 即恢复后队列；会话 / 已推历史
  走 electron-store（**异步 IPC**）——所以恢复整体是 async，由上面 FM 页的 `await` 收口。

**进入 FM 页：在场则续播，不重开**（`pages/heartbeat` 挂载 effect 先查 `isSessionLive()`）：
FM 是路由页，切去别的歌单再点回来 = 组件重新挂载，无条件 `start()` 会清缓存 + 重置种子轮转 +
整队替换，体感「每次点 FM 都从头开始」。判定复用续供那套「队列里还有没有本会话 id」
（`sessionLive()`，`maybeTopup` 与页面共用一处定义）——**只看 `active` 不够**：歌单页「播放全部」
直接调 `playList` 没走 `stopAndReplace`，`active` 会短暂仍为 true，光看它会「盯着 FM 页、耳朵里
放着别的歌单」。**与下面已否的「进入 FM 不打断当前播放」不冲突**：不在场时进入照样打断重开，
只有在场才续播。

**Phase 2 网易腿隔离容错（敢默认开、无开关的前提）**：
- 端点 `music.163.com/api/v1/discovery/simiSong?songid=` 实测**未签名 / 无 cookie
  直通**（curl 验证过，ACG / 日系覆盖极好）。
- 去程只用清洗后的歌名搜网易，**不带 B站 UP 名**——搬运号名不是歌手，带上反而
  降低匹配率。
- 全程 `try/catch` 失败即静默返 `[]`；回程撞 `-412/-352` → 熔断 10min；回程内存
  缓存「歌名→候选」；`5s` 软超时；每次构建只 `3 种子 × 3 首`。
- **与 B站 session 完全隔离，网易请求触发不了 B站风控；对 Phase 1 零侵入**（原两
  条腿一行没动），整条挂掉只是退回 Phase 1 效果。

### 已经否过的方向

- **自己造协同过滤表**：需海量听歌行为，造不出，借网易的表。
- **因 THE FIRST TAKE 一个案例把所有录音棚现场拉黑**：用户明确不要一刀切，改成
  精准黑名单词（`直拍 / 对比 / 测评 / 同台 / THE FIRST TAKE` 等）。
- **只采样红心前 6 首当种子**：用户要求覆盖全部红心 → 改种子轮转。
- **二度扩展种子拿「FM 已服务候选」**：种子取自 FM **自己推过**的歌（可能含用户划
  走 / 不喜欢的），且只带 bvid → 只喂看了又看（只跨 UP），时间久了顺着 FM 自产往外
  漂、越来越不像私人 FM。改用**用户认可的歌**当种子（见下两条演进）。
- **靠「读收藏夹」拿新收藏**：读本地 `folderItems` 只能拿到本地歌单，**读不到 B站在线
  收藏夹**（那批在 B站服务器，本地拉要打接口 + 有风控成本）。改成**记录收藏动作本身**
  （收藏弹窗 confirm 时记，不看收进哪个歌单）→ 本地 / 在线都覆盖，纯本地零网络零风控。
  用户点破：「不需要思考用户收藏了哪个歌单，只要记录收藏了哪首歌」。
- **记「所有收藏动作」**：进一步收窄成**只记「FM 播放时收藏 FM 推荐过的歌」**（`noteFavoriteFromFm`
  双门槛：active + `servedBvids.has`）。这样是个强化闭环——FM 推、用户认可、再顺着找同类；
  非 FM 推的歌 / 非 FM 时段的收藏都不算。也不再读现有本地收藏当种子（那些不是 FM 推的）。
- **进入 FM 不打断当前播放**：用户明确要「进入即打断、直接开始」。注意这说的是
  **不在场**时进入；**在场**（切走看了看又回来）不重开是另一回事，见上「进入 FM 页：
  在场则续播」，两者不矛盾。
- **「换一批」按钮 / ❌不喜欢按钮**：换一批去掉（本来就在不停加新歌）；❌改「上
  一首」，红心歌跳过不算负反馈、非红心歌跳过才算。
- **Phase 2 加 `enableExternalSimilar` 开关**：用户明确不要开关，默认开、用户无感。
- **接 Last.fm 相似表**：用户明确说不做，只网易。

### 边界情况

- **分P**：曲库 item 自带其 P，切走时整队替换各带各 P，按点击项 bvid 旋转到起点。
- **网易腿全挂**：静默空，退回 Phase 1 两条腿，FM 照常播。
- **Android**：`getNeteaseSimilarSongs` 为 noop（返 `{}`），这条腿静默空，不影响。
- **改了 electron 主进程 / preload**（channel + handler + preload）：实机验证需
  **完整重启 `pnpm dev`（非 HMR）**，新 IPC 才加载。
- **在线学习反馈**（本程内切歌 / 听完实时调权）留作后续阶段——现只做到「红心里
  UP 出现频次」排序，要动 play-list 内部，不在本阶段产物内。
- **重启后**：见「关键决策 · 会话跨重启」。已做成「接着放」。

## ✍️ 当前在做

实机整体实测。完整重启 `pnpm dev` → 进「私人FM」→ 开 DevTools：

1. Console 看 `[heartbeat]` 候选池有没有**跨作品 / 跨歌手**的歌（网易腿生效）；
   Network 看有没有 `music.163.com` 请求、有没有 `-412/-352`（撞风控会熔断）。
2. 进合集 / 系列，点**单曲** + **播放全部**，确认都切走 FM、队列换成该歌单。
3. 回归：FM 中全网搜索点歌仍是**插播、不停 FM**；不在 FM 时合集 / 系列点歌正常插播。
4. 会话在场判定：FM 中切去别的歌单**只看不点** → 回 FM **接着放不重开**；在别的歌单
   **点了歌**再进 FM → **重开一轮**。
5. 跨重启：FM 放着 → 关程序 → 重开 → 进 FM **接着放不重开**，且**播到队尾还能续上新歌**
   （续供订阅重挂成功的判据；Console 应有第二组 `[heartbeat]` 输出）。反例回归：退出前
   先点别的歌单再关，重开进 FM 应**重开一轮**。
