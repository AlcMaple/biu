# 播放卡顿 — PCDN 换源容错复盘(2026-06-12)

> 本文记录一次完整的排查与修复过程:症状、诊断路径(含走过的歪路)、最终方案、刻意没做的事、
> 以及日后如果问题复发的升级路线。**动手改播放/取流相关代码前,请先读完本文,避免重复踩坑。**

## 一、症状

用户报告:收藏夹里某一首歌(B 站视频)播放异常,表现为三种形态随机出现:

1. 一直卡着完全播不出来;
2. 播放一小会儿后卡住不动;
3. 直接报错,被"自动跳过坏曲"逻辑跳到下一首。

同一首歌在 B 站官网播放完全正常,视频本身没有下架。另一个佐证细节:播放栏显示的总时长
(05:17)与收藏夹列表里的时长(04:01)对不上。

## 二、诊断过程(含歪路,按时间顺序)

### 歪路 1:先入为主怀疑 cid 失效

最初的探索方向是"缓存的 cid 过期/视频被换源导致取错流",因为时长不一致很像取错了分集。
**这个方向是错的**——后来证明时长不一致只是坏流的副作用(audio 元素从损坏的流里解码出了
错误的时长),不是 cid 问题。

> **教训:别先理论后证据。本项目 renderer 的 `log.error` 会写进主进程日志
> `%APPDATA%\biu\logs\main.log`,里面有播放失败时的完整现场(失败的 src、MediaError 错误码)。
> 排查播放问题第一步永远是看这个日志,一条日志顶半天猜测。**

### 实锤:日志直接给出了答案

日志中的失败记录:

```
音频播放失败，自动跳到下一首 {
  src: 'https://xy120x199x32x152xy.mcdn.bilivideo.cn:8082/v1/resource/...',
  code: 4,
  message: 'MEDIA_ELEMENT_ERROR: Format error'
}
```

- 主机名 `xy…xy.mcdn.bilivideo.cn`(以及 `*.szbdyd.com`)是 B 站的 **PCDN/MCDN 节点**
  (P2P 边缘节点)。这类节点对非官方客户端限速严重,甚至返回坏数据,是第三方 B 站播放器
  社区公认的顽疾。
- `code: 4` = `MEDIA_ERR_SRC_NOT_SUPPORTED`,坏节点吐了无法解码的内容。

### 歪路 2:第一版修复不完整

第一版只改了"取流时优先挑非 PCDN 地址"(见下文修复 ①),以为完事了。用户两个追问暴露了
两个漏洞:

1. **"为什么只有这首歌被针对?"** —— 因为播放地址带 `deadline`(约 2 小时有效期),会缓存在
   播放队列条目上反复使用(`play-list.ts` 中 `isUrlValid` 判定)。坏地址一旦被存下,这首歌
   在有效期内次次用它,看起来就像"被针对"。只改取流偏好,治不了已经缓存的坏地址,也治不了
   "整批候选全是 PCDN"的小概率情况。
2. **"卡住但不报错怎么办?"** —— 坏节点最常见的表现是**不触发 `onerror`、静默挂起**
   (连接不断但不给数据)。只挂在 `onerror` 上的容错对此完全失明,而这恰恰是用户的主诉症状。

> **教训:修播放问题时,把"失败"的三种形态都过一遍——①明确报错(onerror)、
> ②静默挂起(无事件)、③取地址阶段就失败(API 报错)。只覆盖其一的方案是半成品。**

## 三、最终方案(三层,全部已落地)

设计原则:**对齐 B 站官方播放器的容错行为**(坏源自动切换),而不是自创野路子。

### ① 取流时优先正规源 — `src/common/utils/audio.ts`

`collectStreamUrls()`:把 `baseUrl + backupUrl` 去重后按"正规源(upos 等)优先、PCDN 兜底"
排序。PCDN 判定:主机名以 `.mcdn.bilivideo.cn` 或 `.szbdyd.com` 结尾。`getDashUrl` /
`getAudioUrl` 同时返回完整候选列表 `audioUrlCandidates`,随播放条目持久化
(`play-list.ts` 的 `PlayData.audioUrlCandidates`)。

这是**主修复**:日常播放从此不碰 PCDN,卡顿从"必然"降为"小概率"。后面两层都只是保险丝。

### ② 播放失败换源自救 — `src/store/play-list.ts` 的 `tryFailoverAudioSource`

`onerror` 触发时不再直接跳歌,而是:

1. 换一个**没试过的**候选地址,从中断进度续播(不从头播);
2. 候选用尽 → 重新向 B 站取一次新地址再试;
3. 都失败 → 走原有的"提示 + 跳下一首"兜底。

### ③ 卡死看门狗 — 同文件,`armStallWatchdog`

监听 `waiting` / `stalled` 事件,**8 秒**内播放进度毫无推进 → 视为坏源,走与 `onerror`
相同的统一失败处理(`handlePlaybackFailure`)。期间有进度推进则视为网络慢,重新计时观察。
这一层专治"静默挂起",即歪路 2 中的漏洞。

### 防御性约束(防限流、防死循环,改动时不得削弱)

| 约束 | 实现位置 |
|---|---|
| 每个候选地址每首歌只试一次 | `failoverTriedUrls` 集合 |
| "重新取地址"每首歌只允许一次 | `failoverRefreshed` 标记 |
| 切歌即作废:自救绑定发起时的 playId,不符立即收手 | playId 守卫 |
| 用户切歌产生的 ABORTED(code 1) 不算失败 | `onerror` 入口过滤 |
| 整列表轮一圈全失败 → 停止并提示,不无限 next() | `consecutiveErrorCount` |
| 暂停/恢复播放时看门狗正确清除/重挂 | `onpause` / `onplaying` |

**限流账本**:换备用地址是碰 CDN(非 API);动用 API 的只有"重新取地址",每首限一次。
连续 N 首坏歌 = 额外 N 次 API 调用,与用户手动连点 N 首歌等价,距离风控阈值差几个数量级。
另外失效检测(打开收藏夹时后台批量查询并打标,见 `src/common/utils/fav.ts`、
`local-fav-items.ts`)会让已知失效项在播放入口直接被绕过,根本进不了自救流程。

### 顺带修复:本地收藏夹偶发保存失败(EPERM)

日志里同时发现 `local-fav-items.json` 写入偶发 `EPERM`(本地数据在"文档\Biu",会被
OneDrive/杀毒占用,原子写的 rename 被拒后直接丢弃本次写入)。已在 `electron/ipc/store.ts`
加 `setStoreWithRetry`:0.2/0.5/1/2 秒退避重试共 5 次;按 store 维护写入序号,重试期间来了
更新的数据则放弃旧数据,防止旧值回写覆盖新值。

## 四、刻意没做的事(及理由)

1. **没有预加载下一首的播放地址**:切歌时取地址只需零点几秒,慢的从来是坏节点传数据而非
   API,预加载收益极小。
2. **没有做"双播放器无缝交接"**(官方那种两个 media 元素同时缓冲、交叉切换):工程量大、
   引入新 bug 面,在病根(PCDN)已除的前提下收益很小。
3. **没有把看门狗调得更激进**(<8 秒):太短会在网络真慢(WiFi 信号差)时把好源误判成坏源,
   极端情况下导致整列表被误跳。8 秒是误判风险与恢复速度的折中。
4. **没有给"取地址阶段就失败"(无法获取音频播放链接 toast)加自动跳歌**:该失败也可能是
   整网断开,自动跳会在断网时瞬间扫穿整个列表,维持现状(提示并停住)更安全。

## 五、日后复发的升级路线(重要:先读这段再动手)

**触发条件**:新版本实际使用中,仍**反复**出现"播到一半卡 8 秒再恢复"。偶发一两次不算,
那是看门狗在正常工作。

**升级原则(用户明确要求,务必遵守)**:

> 不要为了解决问题采取暴力破解。自己拍脑袋的措施不一定最好,**有前人走过的路一定先借鉴**
> ——前人这么走必有他的道理。动手前先研究参考实现,再决定方案。

**按激进程度排序的候选手段**(逐级使用,前一级无效才上后一级):

1. **调参**:看门狗 8 秒 → 5~6 秒(一行常量 `STALL_TIMEOUT_MS`)。先确认日志里卡死时长
   分布再调,不要盲调。
2. **缓冲余量预警**:播放中周期性检查 `audio.buffered` 末端与 `currentTime` 的差值,
   余量持续萎缩且低于阈值(如 5 秒)时,**在缓冲耗尽前**提前换源——把"卡住后恢复"变成
   "无感切换"。参考依据:hls.js / dash.js 的 ABR 与 stall 检测都基于 buffer health 而非
   等报错,这是流媒体播放器的成熟惯例。注意:直接改 `audio.src` 仍会有短暂中断,此手段把
   中断从 8 秒缩到 ~1 秒以内,但不为零。
3. **双元素无缝交接**:第二个隐藏 audio 元素预载新源、对齐进度后切换输出。参考依据:
   B 站官方网页播放器(基于 MSE 的分片级容错与 CDN 切换)、以及各 gapless 播放实现。
   工程量大,是最后手段;真到这步,优先研究官方播放器与 hls.js 的分片重试/多 CDN 策略,
   而不是自创交接逻辑。

**诊断入口备忘**:

- 日志:`%APPDATA%\biu\logs\main.log`(自动换源会留 `log.warn`,跳歌会留 `log.error`,
  含 src 与错误码);
- 看失败 src 的主机名:`mcdn`/`szbdyd` = PCDN 又漏网了(检查 `isPcdnUrl` 的判定是否要
  扩充域名);`upos-*` = 正规源也出问题,另查;
- 播放地址有效期约 2 小时(`deadline` 参数,`isUrlValid` 判定),复现问题时注意缓存地址
  是否还在有效期内。
