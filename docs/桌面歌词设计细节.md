### 一、 核心排版与 CSS 细节 (视效基石)

* **统一容器与等比间距**：必须将主歌词和副歌词放入**同一个 Flex 容器**中，利用原生的 `gap` 属性控制两者的间距。这样在进行整体缩放时，间距才会等比例缩小，避免出现“容器变大、歌词间距被撕裂”的问题。
* **顶部锚点防漂移**：歌词容器的缩放原点必须设置为 `origin-top`（即 `transform-origin: top center`）。这能保证在垂直拉伸窗口时，歌词始终紧贴上方的工具栏，而不会因为 `justify-center` 的特性跑到屏幕正中央去。
* **提前占位防抖动 (1px Shift)**：为了防止鼠标移入容器添加边框时导致的 1px 挤压抖动，容器在默认状态下就必须挂载一个透明边框（`border border-transparent`）。悬浮时只改变边框颜色，绝不改变边框宽度。
* **内侧缩放句柄**：放弃原生的 `resize` 属性，采用 8 个绝对定位的隐形 div 实现全向缩放。这些句柄的坐标必须设为 `0` 紧贴容器内侧，以防止被最外层的 `overflow-hidden` 裁切掉导致无法拖拽。

### 二、 动态缩放算法 (性能与逻辑核心)

* **GPU 加速优先**：绝对不要使用 JS 监听拖拽去动态修改 `font-size`，这会引发疯狂的 DOM 重排导致卡顿。必须使用 `transform: scale()` 进行 GPU 层面的按比例缩放。
* **双轴动态感知**：缩放比例必须同时考量宽度和高度。计算公式为 `Math.min(可用宽度/原始宽度, 可用高度/原始高度, 1)`。
* **Padding 吞噬空间陷阱**：在计算可用高度（`availableH`）时，务必警惕容器内边距（Padding）占用的绝对物理像素。例如，上下 Padding 总计 72px 时，如果窗口最小高度设为 80px，留给歌词的高度只剩 8px，会导致缩放比例跌至 1% 甚至肉眼不可见。
* **比例兜底机制**：为了防止极端缩小情况下的不可见问题，必须给最终的缩放比例设置一个下限（例如 `finalScale = Math.max(0.35, finalScale)`），确保歌词最小也有个保底尺寸。

### 三、 事件监听与状态管理 (交互陷阱)

* **锁定状态的鼠标穿透**：点击“锁定”后，给外层容器添加 `pointer-events: none` 是灵魂所在，它允许鼠标点击事件穿透到桌面底层。
* **虚拟悬浮检测**：由于 `pointer-events: none` 会导致 CSS 的 `:hover` 彻底失效，必须在全局 `window.addEventListener('mousemove')` 中，通过获取容器的 `getBoundingClientRect()` 并比对 `e.clientX / e.clientY` 来判断鼠标是否进入了歌词区域，从而动态显示“解锁按钮”。
* **事件顺序阻断问题**：在执行“锁定”动作时，必须**先清除**悬浮背景状态（如调用 `setHoverState(false)`），**再将** `isLocked` 标志位设为 `true`。如果顺序反了，清除悬浮状态的逻辑会被 `if (isLocked) return;` 拦截，导致半透明背景永远卡在屏幕上。
* **即时 UI 响应补偿**：点击“锁定”按钮的瞬间，鼠标必然处于容器内部。此时不要等待鼠标下一次移动去触发虚拟悬浮，而是应该在锁定逻辑中**强制将解锁按钮的透明度设为 100**，消除视觉延迟。
